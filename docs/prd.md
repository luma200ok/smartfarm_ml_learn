# 📋 PRD — 스마트팜 재배 도우미 AI (SmartFarm Cultivation Assistant)

> **한 줄 컨셉:** 센서는 *환경 숫자*를 보여주지만, 내 AI는 *지금 작물에 뭘 해줘야 하는지*를 알려준다.
>
> 작성일: 2026-06-18 · 상태: 기획(초안) · 학습 프로젝트 (ML → DL → LLM)
> 위치: `toy_project/solo/smartfarm_ai` · 참조: `toy_project/solo/weather_ml`

---

## 1. 배경 (Why)

- 스마트팜 대시보드는 많지만 대부분 "현재값 표시"에서 멈춤 (온도 28도, 습도 60%, 토양수분 35%).
- 재배자가 진짜 원하는 건 숫자가 아니라 **행동 지침** — "물 줘야 해? 환기해? 이 잎 반점은 병이야?"
- 단순 임계값 알림(`온도>30 → 팬 ON`)은 누구나 만듦 → **차별점은 "예측 + 진단 + 처방"**: 환경 흐름을 학습하고, 잎 사진으로 병을 진단하고, 사람 말로 조치를 안내.

## 2. 문제 정의 (Problem)

| 기존 | 문제점 |
|---|---|
| 스마트팜 대시보드 | 현재 숫자만 줌, 해석·대응은 사람 몫 |
| 임계값 자동제어 | "지금 넘었다"만 알지, 미래/원인은 모름 |
| 병해충 진단 | 전문가 눈에 의존, 초보는 놓침 |

→ **결론:** "센서값 모니터링"이 아니라 **"재배 의사결정 보조"** 로 풀어야 의미 있고, ML→DL→LLM 세 단계가 각자 명분을 가짐.

## 3. 목표 (Goal)

- **핵심 목표:** 환경 데이터 + 잎 사진을 받아, 작물에 필요한 조치를 자연어로 안내.
- **학습 목표:** 하나의 도메인(스마트팜)으로 ML → DL → LLM 풀스택 경험.
- **차별화 목표:** **멀티모달(정형→이미지→언어)** 로 "DL이 ML 재탕"이 아닌, 단계마다 새 능력이 생기는 프로젝트.

## 4. 타겟 사용자 (Who)

- 1차: **스마트팜 초보 / 베란다·옥상 텃밭러** (전문 지식 없이 작물 키우는 사람)
- 확장(니치): 🍅 토마토 시설재배 농가 / 🌶️ 특정 작물 전용 / 🏫 스마트팜 교육용 / 🪴 식물 집사(반려식물)
- 니치를 좁힐수록 차별화 ↑ (예: "토마토 시설재배 전용 — 환경예측 + 잎 병해충 진단 + 처방")

## 5. 차별화 전략 (Differentiation) ⭐

> 겉모습(스마트팜 대시보드)은 흔하다. **멀티모달 깊이(ML→DL→LLM)** 와 **처방까지** 로 차별화한다.

### ① 멀티모달 단계 점프 (가장 강력)
```
정형 센서(ML 분류·예측) → 잎 이미지(DL 비전 진단) → 자연어 처방(LLM)
→ 각 단계가 "앞 단계가 못 하던 일"을 함 (재탕 아님)
```
- ML은 이미지를 못 읽음 → **DL(CNN)이 병 진단** → 명분 명확
- CNN은 라벨만 뱉음("탄저병") → **LLM이 원인·조치 설명** → 명분 명확
- **이 점프 구조가 이 프로젝트의 시그니처.** (출근비서는 ML도 DL도 시계열 예측이라 점프가 약함)

### ② 진단 → 처방 (행동 지침)
```
"탄저병 87%" (숫자) → "탄저병이 의심됩니다. 감염 잎을 제거하고
습도를 낮추세요. 확산 방지 위해 통풍을 늘리세요." (처방)
```
- 효과: LLM이 의미 있어짐(라벨 재탕 X), 사용자가 바로 행동 가능.

### ③ 단일 작물로 시작 → 다작물 확장 (전이학습)
- 작물 하나(**토마토**)로 ML·DL·LLM 파이프라인을 **깊게** 완성 → **전이학습으로 작물 1개씩 확장**(딸기·파프리카·고추…) → 데이터 있는 작물 전부 관통.
- 초심자(여러 작물 키움) 모토 충족 = **단일 깊이 + 다작물 너비** 둘 다. 최종 앱: "작물 선택 → 잎 사진 → 진단+처방".
- 📐 설계 결정 근거 → [decisions.md](decisions.md) (ADR-002).

### 차별화 한 줄
> "센서는 환경을 보여주지만, 내 AI는 **작물에 뭘 해줘야 할지**를 알려준다."

## 6. 핵심 기능 (Features)

| 기능 | 설명 | 단계 |
|---|---|---|
| 환경 적합도/이상 예측 | 센서값 → 작물 적합도 분류 / 이상 탐지 | Phase 1 (ML) |
| 환경 시계열 예측 | 지난 N시간 → 향후 환경 추세 예측 | Phase 2 (DL-시계열) |

| 재배 가이드 RAG | 농사로 재배 매뉴얼 검색 결합 | Phase 3 |
| 알림 | 이상/병해 감지 시 자동 알림 | Phase 3 |
| 📊 발표자료 | 매 단계 5장(문제→데이터→방법→결과→배운점) | 전 단계 |
| 🚀 Streamlit 배포 | 매 단계 데모 앱, 점진 확장 | 전 단계 |

### 처방 로직 (예시)
| 조건 | 안내 문구 |
|---|---|
| 토양수분 낮음 | 💧 관수 필요 — 물 주세요 |
| 온도 상한 초과 예측 | 🌡️ 곧 고온, 환기·차광 준비 |
| 습도 높음 + 병해 진단 | 🍃 통풍 늘리고 감염 잎 제거 |

| CO₂/일사 부족 | ☀️ 보광 또는 환기 조정 |

## 7. 단계별 범위 (Scope / 단계별 MVP)

### Phase 1 — ML (정형 센서 → 작물·환경 판단)
- **데이터:** Kaggle Crop Recommendation (N·P·K·온도·습도·pH·강수 → 작물) — 입문용 깨끗한 CSV
  - 대안/확장: 스마트팜코리아·공공데이터포털 시설원예 환경 CSV
- **입력:** 토양 N·P·K, 온도, 습도, pH, 강수 → **정답:** 적합 작물(분류) / 환경 이상 여부
- **모델:** LogisticRegression/DecisionTree → RandomForest → XGBoost (쉬운 것부터)
- **기술:** scikit-learn, pandas
- **📊 발표:** EDA 시각화(상관 heatmap·분포) + 모델별 정확도 비교 차트 (5장)
- **🚀 배포:** Streamlit 데모 (슬라이더로 환경값 입력 → 작물 추천/적합도 출력)
- **산출물:** 코드 + 발표자료 + Streamlit 앱
- **완료 기준:** 예측 + 정확도(Accuracy/F1) 산출, EDA 완료, **Streamlit 데모 작동**

### Phase 2 — DL (시계열 + 비전) ⭐ 차별화 핵심
> **왜 DL?** ① 환경은 순서가 중요(LSTM) ② **잎 사진은 ML로 못 읽음 → CNN 필수**. 비전이 들어와 ML→DL 점프가 진짜가 됨.

- **2-a 시계열(LSTM/GRU):**
  - 입력: 지난 N시간 환경 시퀀스 → 출력: 향후 환경(온습도) 추세
  - 데이터: 스마트팜코리아 시계열 / 공공데이터 / (없으면) 가상 센서 시뮬레이션
- **2-b 비전(CNN):**
  - 입력: 잎 사진 → 출력: 질병 분류(건강/탄저병/잎곰팡이 등)
  - 데이터: **PlantVillage**(Kaggle, 가입 없이 바로) / AI Hub 농작물 병해충(한국 데이터)
- **기술:** PyTorch(torchvision) 또는 TensorFlow/Keras, 전이학습(ResNet 등)
- **📊 발표:** 학습곡선 + "ML vs DL" + 혼동행렬/예측 샘플 (5장)
- **🚀 배포:** Streamlit 확장 (Phase 1 앱에 **사진 업로드 → 병 진단** 추가)
- **산출물:** 코드 + 모델 저장(.pt/.h5) + 발표자료 + 업그레이드된 Streamlit 앱
- **완료 기준:** 비전 분류 정확도 산출, **Streamlit에서 사진 올리면 진단 결과 표시**

### Phase 3 — LLM (통합 재배 도우미)
> **핵심:** LLM은 진단을 "하지" 않는다. Phase 1·2 결과를 받아 **원인·조치를 사람 말로** 안내한다.
- **재료:** 환경 예측(ML/DL-시계열) + 병 진단(DL-비전) + 재배 가이드(RAG)
- **동작:** 센서·사진 입력 → 예측+진단 → LLM에 전달 → 자연어 처방 생성 → 화면/알림
- **기술:** Claude API(function calling) + 재배가이드 RAG(농사로) + 알림봇(텔레그램/Discord)
- **📊 발표:** 아키텍처 다이어그램 + 실제 처방 데모 영상/스크린샷 (5장)
- **🚀 배포:** Streamlit 대시보드(환경·진단·처방 한 화면) + 알림봇 실서비스
- **산출물:** 코드 + 발표자료 + Streamlit 대시보드 + 작동하는 알림봇
- **완료 기준:** 사진+센서 넣으면 자연어 처방이 나오고, 이상 시 알림 자동 발송

## 12. 레포·배포 전략 (Repo & Deploy) ⭐ 결정됨

> **결정(2026-06-18): 모노레포 1개 "깊이" + 단계별 개별 배포.** (레포 3개 분리 ❌)

### 왜 1개 깊이인가
- 결정권자(기술 면접관)는 "**완주·깊이**"를 봄 — 튜토리얼 3개보다 굴러가는 찐 1개가 희소.
- HR 키워드 스캔(ML/DL/LLM)은 **1레포 README에도 다 노출** → 폭도 동시 충족.
- 세 단계가 **톱니처럼 물려** 돎(데이터·모델·유틸 공유) → 분리하면 연결을 인위적으로 끊어야 함.

### 톱니 ↔ 개별 배포 (둘 다 됨)
- **공유(물림):** `data/` · `models/`(dl 저장→llm 로드) · `src/common/`
- **분리(개별 배포):** `app/phase1_ml.py · phase2_dl.py · phase3_llm.py` → **Streamlit Cloud 앱 3개로 따로 배포** → 라이브 URL 3개
- 과제의 "**3개 배포 완성**" 요구를 모노레포에서 그대로 충족.

### 포트폴리오 노출 (dopacheck 방식)
- `README.md` = 허브 → 단계별 `docs/ml.md · dl.md · llm.md` 하이퍼링크로 유도
- 각 .md = 문제→데이터→방법→결과→배운점 (단계별 기록 = 포트폴리오)
- 본인 계정 포트폴리오 배치: ① dopacheck 미니 협업 → ② **SmartFarm AI(이 레포, 3 데모 링크)** → ③ dopacheck 파이널 협업

### ⚠️ 주의 — 과제 제출 형식이 "레포 3개 강제"일 경우만 차선
- 그땐 3레포 + umbrella README 크로스링크 + **ml 레포 살 붙이기**(EDA 깊이+이상탐지로 튜토리얼 탈피).
- 그 외엔 위 모노레포 깊이 전략 유지.

### 🔁 폴백 — 3레포가 필요해지면 "분리"가 아니라 "추출"
- 처음부터 쪼개지 말 것. **모노로 끝까지 개발 → 각 단계 끝나는 시점에 그 패키지만 떼서 레포로 올림** (`git subtree split -P src/ml -b ml-repo` → 새 레포 push).
- 평소엔 톱니로 물려 편하게 개발하고, 마무리에만 추출 → 비용 0으로 3카드도 확보. (과제 강제 아니면 이 방식이 최선의 폴백)
