# 🧭 설계 의사결정 기록 (ADR)

> smartfarm_ai 프로젝트의 주요 설계 결정과 **그 이유**. 나중에 "왜 이렇게 했지?"를 위해 남긴다.
> 작성: 2026-06-25 — Phase 2(DL) 착수 전 **교통정리**.
> *바닥(데이터·아키텍처)이 안 맞으면 위가 안 맞는다 → 코딩 전에 정체성을 확정한다.*

---

## ADR-001 — Phase 1(노지 데이터)은 ML 입문 단계로 보존, 갈아엎지 않음

**배경**
- Phase 1 ML = Kaggle Crop Recommendation (인도 **노지**, N·P·K·**강수량**, 작물 22종).
- 그런데 PRD 비전은 **스마트팜(시설·양액재배)**. 강수량·토양 NPK는 시설재배와 측정 항목이 다르다.
- 22종(rice·jute·cotton·coffee…)도 시설 주력작물(토마토·딸기·파프리카)과 거리가 멀다.
- PRD 자체에 긴장 존재: §5③ "단일작물 일관성" ↔ §11 "Phase 1은 무관".

**결정**
- Phase 1(노지 22종)은 **ML 입문 완성품으로 동결·보존**. 옛 레포 `smartfarm_ai`에 기록용으로 남긴다.
- 포지셔닝: "범용 ML 입문 → 스마트팜 특화로 성장"의 **1단계**.

**이유**
- 이미 배포·문서화된 산출물(RF 99.5% · Streamlit · 수행내역서). 학습 가치(ML 5단계·평가·배포)는 데이터와 무관하게 유효.
- 성장 서사로 포트폴리오에 오히려 +.

**기각한 대안**
- Phase 1을 스마트팜 데이터로 즉시 재구축 → 완성품 재작업 비용 큼. (정말 필요하면 새 레포에서 v2로)

## ADR-002 — 단일 작물(토마토)로 시작 → 전이학습으로 다작물 확장

**배경**
- PRD 타겟 = 초심자(여러 작물 키움) → 단일 작물로만 끝내면 "초심자 가이드" 모토와 안 맞는다.
- 그러나 처음부터 N작물 동시 = 클래스 폭발·디버깅 지옥.

**결정**
- 토마토 1종으로 **전체 파이프라인**(데이터로더 → 전이학습 → Grad-CAM → 평가 → Streamlit) 완성.
- 같은 파이프라인 + 전이학습으로 **작물 1개씩 추가**(딸기·파프리카·고추…) → 데이터 있는 작물 전부 관통.
- 최종 앱: **"작물 선택 → 잎 사진 → 진단 + 처방"**.

**이유**
- 전이학습 = 작물 추가 비용 ↓ (ImageNet 백본·코드 틀 재사용 → 작물당 적은 데이터·짧은 학습).
- **"단일 깊이 + 다작물 너비"** = 초심자 모토 + 단일작물 일관성을 **둘 다** 충족.

## ADR-003 — 본 구현은 새 레포에서 진행 (옛 레포는 기록용)

**배경**
- 새 비전(스마트팜 단일→확장) vs Phase 1(노지 입문)은 데이터·정체성이 근본적으로 다르다.
- "바닥(아키텍처)이 안 맞으면 위가 안 맞는다" → 깨끗한 분리가 필요.

**결정**
- 옛 레포 `smartfarm_ai` = Phase 1 ML 입문 **기록용 동결**.
- **새 레포** = 스마트팜 **토마토 단일 → 다작물 확장** 본 프로젝트 (Phase 2~).
- 옛 레포의 자산(문서 체계 · concepts · 코딩 컨벤션 · DL.md/DL_concepts.md) **이전**, 두 README **상호 링크**.

**이유**
- 단일→확장 아키텍처를 처음부터 설계(작물 플러그인 구조 · 작물별 모델 관리).
- 정체성 명확.

**트레이드오프 / 보완**
- 성장 서사가 두 레포로 분절될 위험 → **README 상호 링크**("입문 v1 → 스마트팜 v2")로 보완.
- 인프라 재구축 비용 → 기존 문서·폴더 구조를 **템플릿으로 복사**해 완화.

## ADR-004 — 스마트팜 ML은 "작물 추천"이 아니라 "환경 → 적합도·생육 예측"

**배경**
- 새 레포에서 ML을 스마트팜 데이터로 하려 했으나, 조사 결과 **"환경 → 다작물 추천"** 형태의 스마트팜 데이터는 사실상 없다.
- 이유: 스마트팜(시설)은 보통 **단일 작물**을 키우므로 "여러 작물 중 추천"이라는 과제 자체가 성립하지 않는다. (추천형 데이터 = Phase 1의 노지 Crop Recommendation이 거의 유일)

**결정**
- 스마트팜 ML 과제 = **"단일 작물(토마토) 환경/제어 → 생육·수확량·적합도/이상 예측"**(분류·회귀)로 정의.
- 데이터: AI Hub 지능형 스마트팜 통합데이터(토마토 환경 시계열 20만) · 공공데이터포털 농식품부 스마트팜 빅데이터 API · 농진청 우수농가.
- PRD §6 "환경 적합도/이상 예측"과 정합 (추천 아님).

**함의**
- 노지 Phase 1(환경→**추천**)과 스마트팜 Phase 1(환경→**적합도/생육**)은 **과제가 다르다** → ADR-001(노지 보존)이 더 타당. 고쳐 쓰는 게 아니라 별개 과제.
- 새 레포 우선순위: **Phase 2(잎 진단)부터**. 환경→생육 ML은 환경 시계열 LSTM(Phase 2)에서 일부 실현, 본격 ML은 데이터 확보 후 선택.

## ADR-005 — 구현 순서: DL(잎 진단) 먼저, ML은 그 뒤 탑다운 재설계

**배경**
- ML 단계 정체성(작물 추천 / 적합도 예측 / 데이터 허브·체험)이 계속 재논의되어 결정 피로.
- 반면 DL(토마토 잎 병 진단)은 과제·데이터(PlantVillage)가 명확.

**결정**
- **DL(Phase 2 잎 진단)을 먼저 구축**한다.
- ML 단계는 DL 완성 후 **탑다운으로 재설계**(데이터 허브/적합도/체험 등은 그때 확정). 현재는 **보류**.
- 데이터셋부터 확보 → 전이학습 형태로 작물 확장(→ ADR-002).

**이유**
- 명확한 것부터 만들어 모멘텀 확보. ML 정체성은 DL 결과물을 보고 탑다운으로 정하는 게 더 정확.
- DL 데이터(PlantVillage 토마토)는 즉시, ML 데이터(AI Hub 스마트팜)는 신청·승인 대기 → 병행이 합리적.

**액션**
- DL: PlantVillage 토마토 즉시 확보 → 청크 2-1부터.
- ML: AI Hub 스마트팜 토마토 환경 데이터 **신청만 걸어두고 보류**.

## 📦 데이터 소스 맵 (작물별 · 용도별)

> **전략(ADR-002 연계):** ML 작물 ↔ DL 병해충 작물을 **일치**시킨다. 확장은 "ML·DL 데이터가 **둘 다 있는** 작물"의 **교집합**으로.

| 용도 | 데이터 | 받기 |
|---|---|---|
| ML 환경·생육 (다작물) | 공공데이터포털 **농진청 스마트팜 현장 농가 데이터** (토마토·딸기·파프리카·오이·참외·국화) | CSV 즉시 ✅ (받는 중) |
| DL 병 진단 (이미지) | **PlantVillage 토마토**(즉시, 병9종) → **AI Hub 시설작물 질병진단**(`dataSetSn=153`, 12작물·국내) | 즉시 / 승인 |
| DL 환경 시계열 (LSTM) | AI Hub 지능형 스마트팜 통합데이터(534) 환경 시계열 | 승인 |

> ⚠️ AI Hub **통합데이터(534)**의 이미지는 **생육용(병명 없음)** → 병 진단엔 못 씀. 병 진단은 **시설작물 질병진단 이미지**가 맞음. (용도 혼동 주의)

**작물별 DL 병해충 이미지 가용성** (확장 순서 판단 · ML 농진청 ∩ DL 153 교집합)
- 🍅 토마토: PlantVillage(정상+병9) + AI Hub 153(잎곰팡이병·황화잎말이+정상) — ✅✅ **1순위**
- 🍓 딸기: 농진청 ✅ + 153 ✅ — 확장 가능
- 🥒 오이: 농진청 ✅ + 153 ✅ — 확장 가능
- 🍈 참외: 농진청 ✅ + 153 ✅ — 확장 가능
- 🫑 파프리카: 농진청 ✅ / 153엔 없음(고추는 있음) — △ 국내 병이미지 확인 필요
- 🌼 국화: 농진청 ✅ / 153엔 없음 — ❌ 진단 제외(ML만)

> 153 작물(12종): 가지·고추·단호박·딸기·상추·수박·애호박·쥬키니호박·오이·참외·토마토·포도
> 결론: ML↔DL **둘 다 있는 작물 = 토마토·딸기·오이·참외** → **토마토부터 → 딸기·오이·참외로 확장**.
